---
intent: feature_request
description: Create a normalized feature request file under AGENTS/feature_requests from a user request.
fields: {}
---

Use `feature_request` when the user wants to add or capture a feature request.

Default behavior:

- write the feature request under `AGENTS/feature_requests`
- default to creating a new feature request
- do not ask whether to create a new feature request unless the user explicitly asks you to search, deduplicate, or check for an existing request first
- if the user explicitly asks to see or inspect existing feature requests, run `list_feature_requests`
- do not ask where to put it unless the user explicitly asks for a different destination
- before creating the feature request, ask 2 or 3 short clarifying questions with `ask_human` unless the user explicitly says to just do it or the request is already fully specified
- use the clarifying questions to tighten `problem`, `proposed_change`, and `acceptance_criteria`
- generate `possible_tests` yourself from the request and clarifications unless the user explicitly wants to define the tests
- do not ask where to put it unless the user explicitly asks for a different destination
- if the user explicitly says to drop, cancel, or stop, run `cancel` and then call `done_for_now`

Execution:

1. If the request is not already fully specified, ask 2 or 3 concise clarifying questions in one `ask_human` call.
   Focus on:
   - the exact problem or gap
   - the desired change
   - what should count as done
2. Convert the user request plus answers into these fields:
   - `title`
   - `problem`
   - `context`
   - `proposed_change`
   - `acceptance_criteria`
   - `possible_tests` generated by you from the final request
3. Run the workflow tool:

```bash
printf '%s\n' '{
  "title": "Log model reasoning tokens",
  "problem": "Reasoning token usage is not visible in run logs.",
  "context": "Operators need to inspect hidden-token usage and provider cost behavior after runs.",
  "proposed_change": "Write reasoning token usage to run logs and surfaced summaries when providers expose it.",
  "acceptance_criteria": [
    "Run logs include reasoning token counts when providers expose them.",
    "Runs do not fail when a provider omits reasoning token data."
  ],
  "possible_tests": [
    "Mock a provider response with reasoning token usage and verify it is written to the log artifact.",
    "Mock a provider response without reasoning token usage and verify the run still completes."
  ]
}' | company/workflows/feature_request/tools/create_feature_request/run
```

4. After the tool succeeds, its result will include `terminal.status` and `terminal.instruction`. The next and only valid action is `done_for_now`.

Cancellation:

```bash
printf '{"reason":"User asked to drop the task."}' | company/workflows/feature_request/tools/cancel/run
```

After `cancel` succeeds, its result will include `terminal.status` and `terminal.instruction`. The next and only valid action is `done_for_now`.

Inspection:

```bash
printf '{}' | company/workflows/feature_request/tools/list_feature_requests/run
```

Rules:

- prefer concrete, implementation-ready wording
- keep acceptance criteria observable
- keep possible tests short and engineering-oriented
- ask at most 3 clarifying questions before creation
- keep clarification questions short and specific
- do not ask the user for `possible_tests` by default
- do not ask whether a new feature request should be created by default
- use `list_feature_requests` only when the user explicitly asks to inspect existing feature requests
- do not output plain text directly; use the tool result, then `done_for_now`
